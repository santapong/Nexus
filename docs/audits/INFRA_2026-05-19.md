# NEXUS Infra & K8s Audit — 2026-05-19

Read-only audit of all infrastructure and CI surfaces:

- `docker-compose.yml`, `docker-compose.prod.yml`, `docker-compose.dokploy.yml`
- `backend/Dockerfile`, `frontend/Dockerfile`
- `k8s/base/*.yaml`, `k8s/overlays/{dev,staging,production,prod}/kustomization.yaml`
- `.github/workflows/{ci,deploy,docker-publish,security}.yml`
- `Makefile`, `.env.example`, `deploy/dokploy/`

Reusable skill files for re-running the wider audit suite live in
`.claude/skills/audit-*/`. This document is the infrastructure slice.

---

## Executive summary

The infra layer is **the weakest link in the NEXUS production story.** The
backend application is hardened (the four other audits document its issues), but
the platform that would ship it has gaps that turn a working app into a security
incident the moment a real tenant touches it.

| Production-readiness claim | Reality |
|----------------------------|---------|
| "Enterprise security by default" (CLAUDE.md §3) | K8s pods run as **root**, no `securityContext`, no `runAsNonRoot`, no `readOnlyRootFilesystem` on any workload (`k8s/base/*.yaml`). |
| "Secrets vault upgrade — SOPS-based encrypted secret management" (Phase 5 Track A — DONE) | `k8s/base/secrets.yaml` is a plaintext `stringData` Secret with a real-looking placeholder Postgres password committed in git. SOPS is **not wired** to anything in `k8s/`, `.github/workflows/`, or `scripts/`. |
| "Production-Ready SaaS" (Phase 5 Track A — Track A gate PASSED) | No `NetworkPolicy` anywhere — tenant A's compromised backend pod can hit tenant B's PostgreSQL directly. RLS application bug (covered in main audit §1.2) becomes total isolation failure. |
| "Multi-region / availability zone" | Zero provisions in the overlays. No topology spread constraints, no zone-aware PodAntiAffinity, no PDB on stateful workloads (postgres / kafka / redis). |
| "Backup / disaster recovery" | No PVC backup policy, no WAL archiving config, no Kafka tiered storage. Single replica StatefulSets for postgres and kafka. Data loss on node failure is near-certain. |
| "Deploy to production via `kubectl rollout status deployment/nexus-backend`" (`.github/workflows/deploy.yml:139`) | The deployment is **named `backend`**, and the production overlay applies `namePrefix: prod-` → `prod-backend`. The workflow looks for `nexus-backend`. Rollout status **will never succeed**; production deploys hang for 300 s and time out. |
| "Image pinning" | Every image (incl. `pgvector/pgvector:pg16`, `redis:7-alpine`, `apache/kafka:3.7.0`, `aquasecurity/trivy-action@master`) is pinned by **tag only**, never by `@sha256:` digest. Floating tags = supply-chain vulnerability. |
| "JWT_SECRET_KEY required" (`.env.example:30`, `app.py` security check) | The K8s `nexus-secrets` Secret **does not contain `JWT_SECRET_KEY`**. Backend boot will fail in K8s — unless someone manually injects it, in which case it's outside SOPS/version control. |
| "CI security: pip-audit, bandit, npm audit, gitleaks, trivy all running" (CLAUDE.md §21) | `bandit` and `npm audit` are configured with `|| true` — they **never fail the build**. Trivy in `ci.yml` has no `exit-code: 1` — same. Only `deploy.yml`'s post-build trivy gate is enforced. |

**Total: 9 Critical, 12 High, 11 Medium findings.** Most are 5–30 minute fixes
that have outsized risk reduction. Recommended order in §6.

## 1. Docker images — findings

### Critical

1. **Frontend prod image runs as root**
   `frontend/Dockerfile:23–83` — the `prod` stage is `nginx:alpine`, never adds a
   `USER` directive. Stock nginx runs the master as root and worker as `nginx`,
   but nothing in this Dockerfile pins that and no `securityContext` enforces it
   at the pod level (see §3.1). Any nginx worker RCE → root in container.
   **Fix:** Add `USER nginx` (the alpine image has the user), and ensure the
   conf does not need privileged ports — already uses 80 which requires
   `CAP_NET_BIND_SERVICE` or rebind to 8080. Combine with §3.1.

### High

2. **No image-digest pinning anywhere**
   `k8s/base/postgres.yaml:35` (`pgvector/pgvector:pg16`), `redis.yaml:22`
   (`redis:7-alpine`), `kafka.yaml:23` (`apache/kafka:3.7.0`),
   `docker-compose.yml:3,19,30,110,124,148,156`, every base image in both
   Dockerfiles, and **every GitHub Action** (`docker/setup-buildx-action@v3`,
   `aquasecurity/trivy-action@master`, etc.). A tag-only reference is mutable:
   the upstream maintainer (or an attacker who compromises Docker Hub / GHCR
   credentials) can swap the image bytes under the same tag.
   **Fix:** Pin every image and action by `@sha256:<digest>`. Use Dependabot
   or Renovate with `digest` updates enabled to keep them current.

3. **Backend dev stage shipped in the registry**
   `backend/Dockerfile:1–21` defines a `dev` stage that ships `gcc`, `libpq-dev`,
   uvicorn `--reload`, and bind-mounts `./backend` at runtime via
   `docker-compose.yml:106`. If someone builds without `--target prod` (e.g. the
   default `docker build .` picks the **last** stage which is `prod` — OK), or
   misconfigures the compose target, you ship a dev image with build tools in
   production. The risk is low for the current Compose setup but the dev stage
   is reachable from `docker-compose.dokploy.yml:88` if `target` is omitted.
   **Fix:** Move `dev` into a separate `Dockerfile.dev`, or make the order such
   that `prod` is the last stage and refactor compose to always set `target`.

4. **`pip install` invokes layer caching incorrectly**
   `backend/Dockerfile:14–18` copies `pyproject.toml`, runs `pip install -e .[dev]`
   with `2>/dev/null || true` (silencing every install failure), then copies
   the source and reinstalls. The `|| true` masks dependency errors at build
   time — a missing wheel produces a green build with a broken image.
   **Fix:** Remove `|| true`. Two-step install pattern only makes sense if it
   is a `requirements.txt` + sources split.

### Medium

5. **`apt-get install` without `apt-get clean`**
   Backend Dockerfile lines 6–10 and 31–34: uses `rm -rf /var/lib/apt/lists/*`
   which is correct, but `apt-get clean` is not called. The list-cleanup gets
   most of it; this is minor.
   **Fix:** Append `&& apt-get clean` for completeness; or switch to a
   distroless final stage (Google `distroless/python3-debian12`).

6. **Slim base, not distroless**
   `python:3.12-slim` carries 80+ MB of attack surface that the runtime never
   uses (apt, dpkg, /bin/sh utilities). Distroless cuts this to ~25 MB and
   removes every shell.
   **Fix:** Two-stage build with `python:3.12-slim AS builder` for compilation,
   `gcr.io/distroless/python3-debian12` for the final stage. The healthcheck
   would need to call into Python rather than `curl`.

7. **No `.dockerignore` audit performed**
   Both Dockerfiles do `COPY . .` which depends on `.dockerignore` to keep
   `node_modules`, `.git`, `.env`, and `__pycache__` out. Not audited here.
   **Fix:** Verify `backend/.dockerignore` and `frontend/.dockerignore` exist
   and exclude at least `.git`, `.env*`, `**/__pycache__`, `tests/`, `node_modules`.

### Notes — looks correct

- Backend prod stage has `USER appuser` ✓ (`backend/Dockerfile:46–47`)
- Backend prod has `HEALTHCHECK` ✓ (`:49–50`)
- Frontend has `HEALTHCHECK` ✓ (`frontend/Dockerfile:79–80`)
- Multi-stage build with separate `dev`/`build`/`prod` ✓
- `--no-install-recommends` on apt-get ✓
- `--no-cache-dir` on pip and `rm -rf /root/.cache/pip` ✓
- `TARGETPLATFORM`/`BUILDPLATFORM` build args set for multi-arch ✓

## 2. Docker Compose — findings

### High

1. **Dev compose `JWT_SECRET_KEY` falls back to a known string**
   `docker-compose.yml:88` — `JWT_SECRET_KEY: ${JWT_SECRET_KEY:-nexus-local-dev-only-not-for-production-use}`.
   Combined with `Dockerfile:46` (`USER appuser`) and the fact that the same
   default ends up baked into any `make up`-driven container, anyone with
   shell access to a misconfigured dev host can forge JWTs that the running
   backend accepts.
   **Fix:** Remove the default; require the env var to be set. Compose
   `:?Set JWT_SECRET_KEY` (as `docker-compose.dokploy.yml:123` does) — this
   pattern is correct in the Dokploy file but not in the main one.

### Medium

2. **Redis exposes no `--requirepass` in the main compose**
   `docker-compose.yml:18–27` — Redis listens on `0.0.0.0:6380` with no
   authentication. If a developer publishes the port (it currently is, mapping
   to host 6380), any LAN attacker can connect. The Dokploy compose has
   `--requirepass` (`docker-compose.dokploy.yml:40`) which is correct, but the
   dev compose does not.
   **Fix:** Bind only to 127.0.0.1 (`ports: ["127.0.0.1:6380:6379"]`).

3. **Postgres exposed on `0.0.0.0:5433` with default `nexus_dev` password**
   `docker-compose.yml:7–9` — fine for local dev but a developer running on
   a coffee-shop wifi has an open Postgres on the host.
   **Fix:** Bind to localhost only.

4. **Kafka exposed on `0.0.0.0:9092` with no auth and no TLS**
   `docker-compose.yml:45`. Plaintext listener bound to host. Combined with
   the **dev-mode JWT fallback (above)**, this is an injection vector.
   **Fix:** Localhost-only bind; KRaft + SASL_SSL for prod.

5. **No memory limits on most dev compose services**
   Only `docker-compose.dokploy.yml` defines `deploy.resources.limits`; the
   main `docker-compose.yml` has none, so a runaway Kafka heap or pgvector
   index build can OOM the developer's machine.
   **Fix:** Add `mem_limit:` to all services in `docker-compose.yml` (Compose v3
   accepts the v2 form for non-Swarm runs).

### Notes — looks correct

- All services have `healthcheck` ✓
- `depends_on.<svc>.condition: service_healthy` ✓
- KRaft mode (no ZooKeeper) ✓
- Compose v2 syntax, no obsolete `version:` key ✓
- LLM API keys never default to hardcoded values — `${VAR:-}` empty defaults ✓

## 3. Kubernetes — findings

### Critical

1. **No `securityContext` on any pod or container**
   Searched `k8s/base/` and overlays — zero hits for `securityContext`,
   `runAsNonRoot`, `runAsUser`, `readOnlyRootFilesystem`,
   `allowPrivilegeEscalation`, or `capabilities`. Every workload runs with
   default permissive policy. Default for `pgvector/pgvector` is **root**.
   **Fix:** Add to every container and PodSpec:
   ```yaml
   securityContext:
     runAsNonRoot: true
     runAsUser: 1000
     allowPrivilegeEscalation: false
     readOnlyRootFilesystem: true
     capabilities:
       drop: ["ALL"]
     seccompProfile:
       type: RuntimeDefault
   ```
   And the pod level:
   ```yaml
   securityContext:
     runAsNonRoot: true
     fsGroup: 1000
     seccompProfile:
       type: RuntimeDefault
   ```
   Postgres / Kafka need writable volume mountpoints — pair `readOnlyRootFilesystem`
   with an `emptyDir` for `/tmp` and `/var/run`.

2. **`k8s/base/secrets.yaml` is a plaintext placeholder Secret in git**
   `k8s/base/secrets.yaml:9` — `POSTGRES_PASSWORD: "nexus-prod-password-change-me"`.
   Line 10 repeats it inside `DATABASE_URL`. A real-looking secret committed
   to a public repo trips secret scanners and (worse) people copy-paste it
   into production. SOPS is documented in CLAUDE.md §5 as decided/in-use, but
   `grep -rn 'sops\|age' k8s/` returns zero hits.
   **Fix:** Delete `k8s/base/secrets.yaml`. Replace with a SOPS-encrypted file
   (`secrets.enc.yaml`) under `k8s/base/`, and add a `ksops` or `helm-secrets`
   kustomize generator. Document the AGE recipient pubkey in CLAUDE.md.

3. **No NetworkPolicy anywhere**
   No `kind: NetworkPolicy` exists in `k8s/`. Pod-to-pod traffic is fully open
   within the namespace. The frontend can hit Postgres on 5432 directly. The
   ingress can hit Kafka. A compromised plugin tool inside the backend can
   exfiltrate to any in-cluster service or arbitrary external host.
   **Fix:** Default-deny ingress + egress, then explicit allows:
   - `backend → postgres:5432`, `backend → redis:6379`, `backend → kafka:9092`
   - `frontend → backend:8000`
   - `backend → DNS:53`, `backend → internet:443` for LLM/MCP calls
   - Deny everything else.

4. **`JWT_SECRET_KEY` not in the K8s Secret**
   `k8s/base/secrets.yaml` has only `POSTGRES_PASSWORD`, `DATABASE_URL`, and
   LLM API keys. But `app.py`'s security check (per CLAUDE.md §21) requires
   `JWT_SECRET_KEY` at boot. The backend pod will fail with a 500 on `/health`
   the moment it starts in K8s, unless somebody has manually `kubectl edit
   secret` patched it — at which point the value is outside the GitOps trail.
   Also missing: `ANTHROPIC_API_KEY` etc. are present but empty, which causes
   ModelFactory to fail closed (less bad but still a config bug).
   **Fix:** Add `JWT_SECRET_KEY`, `ENCRYPTION_KEY` (per main-audit Security §1),
   and `KAFKA_HMAC_SECRET` (per Phase 7) to the Secret. Make them all required.

5. **CI deploy targets a non-existent Deployment name**
   `.github/workflows/deploy.yml:102` — `kubectl rollout status
   deployment/nexus-backend -n nexus-staging`. But:
   - `k8s/base/backend.yaml:4` names the deployment `backend`.
   - `k8s/overlays/staging/kustomization.yaml:9` adds `namePrefix: staging-` →
     the deployed name is `staging-backend`.
   - There is no `nexus-` prefix anywhere.
   Lines 103, 109, 139, 140 have the same bug for staging-frontend,
   prod-backend, prod-frontend.
   **Result:** Every tagged deploy hangs on `rollout status` for 300 s and
   exits non-zero. Production rollouts will look like they failed even when
   they succeeded.
   **Fix:** Either use the prefixed names (`staging-backend`, `prod-backend`)
   or remove `namePrefix` from the overlays and use distinct namespaces
   (`nexus-staging`, `nexus-production`) as discriminators.

### High

6. **Single-replica StatefulSets for `postgres` and `kafka`**
   `postgres.yaml:24` and `kafka.yaml:12` — `replicas: 1`. No PodDisruptionBudget
   for either. A node drain or eviction kills the only replica → outage. The
   production overlay (`overlays/production/kustomization.yaml:63–67`) explicitly
   forces postgres to `replicas: 1`.
   **Fix:** Either run a real HA postgres (CloudNativePG / Patroni / Crunchy)
   or document that this is a development-grade setup and add a PDB with
   `minAvailable: 1` so cluster ops know to drain carefully.

7. **PDBs only cover backend + frontend**
   `k8s/base/hpa.yaml:96–117` defines PDB for `backend` and `frontend`. None for
   `postgres`, `kafka`, or `redis` — the workloads where a PDB matters most.
   **Fix:** Add PDB with `maxUnavailable: 0` for all three stateful workloads.

8. **HPA min/max replicas conflict with PDB**
   `hpa.yaml:16` sets `minReplicas: 1` for backend; PDB sets `minAvailable: 1`.
   That means a rolling update can never succeed when only 1 pod is running
   — kubelet refuses to evict the last pod. The HPA needs `minReplicas: 2` at
   least, or the PDB needs `maxUnavailable: 1`.
   **Fix:** `minReplicas: 2` in production; or `maxUnavailable: 1` on PDB.

9. **KEDA / Kafka-lag scaling is commented-out boilerplate**
   `hpa.yaml:46–57` — the lag metric is a comment. CLAUDE.md §15 says
   "Kafka consumer lag metric placeholder ready for KEDA/Prometheus adapter."
   Currently HPA scales on CPU only, which is the **wrong signal** for
   Kafka consumers (CPU stays low while lag explodes).
   **Fix:** Install KEDA, deploy a `ScaledObject` for the backend, scale on
   `kafka_consumergroup_lag > 100`. Without it, the HPA does nothing useful
   under realistic load.

10. **No `imagePullSecrets` configured**
    Workloads reference `nexus-backend:latest` and `nexus-frontend:latest`
    (`backend.yaml:28,37`, `frontend.yaml:22`). When pushed to GHCR as
    `ghcr.io/santapong/nexus-*:*`, the K8s cluster needs an `imagePullSecret`
    or a default ServiceAccount mapping — neither is present.
    **Fix:** Either make the GHCR packages public, or create a
    `kubernetes.io/dockerconfigjson` Secret and reference it via
    `serviceAccount.imagePullSecrets` or per-pod `spec.imagePullSecrets`.

11. **Stale container image references in base manifests**
    `backend.yaml:28,37` and `frontend.yaml:22` use the local development image
    name (`nexus-backend:latest`) — not the GHCR registry path. The
    `deploy.yml:95` workflow sed-replaces `:latest` with `:${VERSION}` but does
    not change the image name itself. So production runs `nexus-backend:1.2.3`
    which is not a valid image reference for any registry.
    **Fix:** Set `image: ghcr.io/${OWNER}/nexus-backend:latest` in the base,
    and let the overlay or sed do the tag substitution.

12. **PVC has no `storageClassName`**
    `postgres.yaml:79–87` and `kafka.yaml:84–92` — `volumeClaimTemplates` lack
    `storageClassName`. Clusters will pick the default, which may be
    `local-path` (no replication) on smaller clusters or `gp2` (no encryption)
    on EKS. No guarantee of encryption-at-rest or backup policy.
    **Fix:** Set `storageClassName: encrypted-ssd` (or env-specific equivalent)
    and document required StorageClass features in `k8s/README.md` (does not
    exist — create it).

13. **`namespace.yaml` does not match overlay namespaces**
    `k8s/base/namespace.yaml:4` declares the `nexus` namespace. But the staging
    overlay (`overlays/staging/kustomization.yaml:4`) uses `nexus-staging`, and
    production uses `nexus-production`. The base namespace resource will be
    re-applied with the staging/production prefix — producing a `nexus-staging`
    namespace via `commonLabels`, but the namespace resource itself is not
    renamed (kustomize does not auto-rename `Namespace` kinds). Net result:
    deploys may attempt to apply the wrong namespace metadata.
    **Fix:** Remove `namespace.yaml` from the base; let each overlay define its
    own namespace resource explicitly.

### Medium

14. **`POSTGRES_PASSWORD` substitution in ConfigMap won't resolve**
    `configmap.yaml:8` — `DATABASE_URL: "postgresql+asyncpg://nexus:$(POSTGRES_PASSWORD)@postgres:5432/nexus"`.
    The `$(VAR)` syntax is **K8s-internal env interpolation**, and it works
    only when a container references both this ConfigMap and a Secret env in
    its envFrom in the right order. Otherwise this string ends up in the
    backend with literal `$(POSTGRES_PASSWORD)`. Inspecting `backend.yaml:41–45`
    — both `envFrom` are referenced, but ConfigMap is listed **first**, so
    `$(POSTGRES_PASSWORD)` is not in scope when the ConfigMap is evaluated.
    Actually env var interpolation in `envFrom.configMapRef.values` doesn't
    work this way at all — it's only in `env[].value`, not in `envFrom`.
    **Fix:** Move `DATABASE_URL` out of ConfigMap into the Secret, or build it
    in code from `POSTGRES_HOST` + `POSTGRES_USER` + `POSTGRES_PASSWORD`.

15. **No `topologySpreadConstraints` or `podAntiAffinity`**
    Backend in production has 3 replicas (`overlays/production:23`), all
    schedulable to the same node by default. A node failure takes out the
    whole backend tier.
    **Fix:** Add `topologySpreadConstraints` with
    `topologyKey: topology.kubernetes.io/zone` and
    `whenUnsatisfiable: DoNotSchedule`.

16. **No `ResourceQuota` or `LimitRange` per namespace**
    No protections against a misconfigured pod requesting 100 CPUs or 100 GiB.
    Multi-tenant SaaS where workspaces share a cluster would be at risk if
    workspaces ever map to namespaces.
    **Fix:** Add `ResourceQuota` (total cpu/memory/pods) and `LimitRange`
    (default + max per container) per overlay.

17. **Kafka KRaft `replication.factor=1` is forced for prod**
    `kafka.yaml:44–48` — `KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR=1`,
    `KAFKA_TRANSACTION_STATE_LOG_REPLICATION_FACTOR=1`,
    `KAFKA_TRANSACTION_STATE_LOG_MIN_ISR=1`. With a 1-replica Kafka, a single
    broker death loses unreplicated data. Production needs RF=3 + min.ISR=2 +
    at least 3 brokers.
    **Fix:** Per-overlay env overrides; production needs 3 brokers + RF=3.

18. **Postgres has no tuned config**
    `postgres.yaml` uses image defaults. `shared_buffers`, `max_connections`,
    `work_mem`, `effective_cache_size`, `random_page_cost` all default. For a
    production agent platform with pgvector workloads, defaults are inadequate.
    **Fix:** Mount a ConfigMap-derived `postgresql.conf` snippet:
    ```yaml
    shared_buffers = 256MB
    effective_cache_size = 1GB
    work_mem = 16MB
    maintenance_work_mem = 128MB
    max_connections = 100
    ```

19. **Ingress lacks TLS**
    `ingress.yaml:18–27` — `nexus-ingress` has no `spec.tls` block. The
    documented host `nexus.local` is plaintext HTTP. cert-manager + Let's
    Encrypt is the standard add-on; not configured here.
    **Fix:** Add `tls:` with `secretName` + `cert-manager.io/cluster-issuer`
    annotation.

20. **Ingress `rate-limit-rpm` annotation absent**
    `ingress.yaml` has body-size and timeout annotations but no
    `nginx.ingress.kubernetes.io/limit-rpm` or `limit-connections`. A scraper
    or DDoS at the ingress will land directly on the backend's HPA, exploding
    cost.
    **Fix:** Set per-IP `limit-rpm: 600` and `limit-connections: 20`.

### Notes — looks correct

- All Deployments and StatefulSets have CPU + memory requests AND limits ✓
- Liveness, readiness, AND startup probes on backend ✓ (`backend.yaml:53–74`)
- Readiness + liveness on every other workload ✓
- `RollingUpdate` strategy with `maxUnavailable: 0, maxSurge: 1` for backend ✓
- `preStop` lifecycle hook on backend for drain ✓ (`:75–78`)
- PDB defined for backend + frontend ✓ (`hpa.yaml:96–117`)
- HPA `behavior` policies (scale-up/scale-down stabilization windows) ✓
- StatefulSets use headless Service (`clusterIP: None`) ✓
- `kustomize` overlays present for dev/staging/production ✓
- Backend has Alembic init container ✓ (`backend.yaml:25–34`)

## 4. CI / CD — findings

### Critical

1. **Production `kubectl rollout status` targets wrong name**
   Covered in §3.5. Same fix.

### High

2. **`aquasecurity/trivy-action@master` not pinned**
   `.github/workflows/ci.yml:252`, `security.yml:82`, `deploy.yml:66`. A
   compromised maintainer or supply-chain attack on the action repo silently
   ships malicious code into the pipeline.
   **Fix:** Pin every action by SHA: `aquasecurity/trivy-action@<40-char-sha>`.
   Use Dependabot's `package-ecosystem: github-actions` to keep them current.

3. **`bandit` and `npm audit` are non-blocking**
   `ci.yml:166` — `bandit ... || true`. `ci.yml:192` — `npm audit ... || true`.
   `ci.yml:255` — Trivy in CI has no `exit-code: 1`. The pipeline reports
   security findings, but **does not fail on them**. The "all running" claim
   in CLAUDE.md §21 is technically true but misleading.
   **Fix:** Remove `|| true` and set `exit-code: 1` on the CI trivy step.
   Keep `--severity-level high` to avoid noise.

4. **Trivy CI scan has no `exit-code: 1`**
   `ci.yml:251–256` — only `severity: CRITICAL,HIGH` and `format: table`. Build
   passes regardless of findings.
   **Fix:** Add `exit-code: 1`. The deploy.yml version is configured
   correctly (`deploy.yml:71`) — copy that.

5. **Secrets-scan job uses unpinned third-party action**
   `ci.yml:202` — `gitleaks/gitleaks-action@v2`. `security.yml:58` —
   `trufflesecurity/trufflehog@main`. Both unpinned; same supply-chain risk.
   **Fix:** Pin by SHA.

6. **CodeQL not enabled on the `ci.yml` workflow**
   `security.yml:95–113` defines CodeQL, but it runs only on `push:main`,
   `pull_request:main`, and weekly. A long-lived feature branch can carry
   CodeQL-detectable bugs without ever being scanned until merge.
   **Fix:** Add `branches: ['**']` to `security.yml` push trigger or run on
   every PR regardless of base.

7. **No SBOM generation**
   No `syft` or `cosign attest --type spdxjson` step. For production SaaS,
   shipping without SBOMs blocks SOC2 / NIST SSDF.
   **Fix:** Add a `syft` step in `docker-publish.yml`; attach as a release
   asset and `cosign attest` it to the image.

8. **Image signing not configured**
   No `cosign sign` or `docker trust sign` step. Pulled images can be swapped
   by a registry compromise.
   **Fix:** Configure keyless signing via Sigstore (`cosign sign --yes
   ghcr.io/...@digest` with `id-token: write` permission).

### Medium

9. **`docker-publish.yml` runs on every push to `main`**
   Lines 4–5. That includes README-only commits — wastes CI minutes and
   pollutes the registry with `latest` churn. Combine with branch-protection +
   path filters.
   **Fix:** Add `paths:` filter to ignore `**/*.md`, `docs/**`, `.github/**` (except
   the workflow itself).

10. **`deploy.yml` smoke test queries `loadBalancer.ingress[0].hostname`**
    `deploy.yml:109` — works on AWS (hostnames) but not on GCP (IPs) or any
    bare-metal/MetalLB cluster which uses `.ip`. The smoke test will fail
    silently on most setups.
    **Fix:** Try both `.hostname` and `.ip`, or use an Ingress endpoint
    instead of the LoadBalancer Service.

11. **`deploy.yml` lacks rollback on failed smoke test**
    Lines 105–110 — if the smoke test fails after `kubectl apply`, the bad
    deployment stays live. No automatic `kubectl rollout undo`.
    **Fix:** Add a `if: failure()` step that runs `kubectl rollout undo
    deployment/...`.

12. **`security-frontend` job is missing `needs:` dependency on lint**
    `ci.yml:175–192` — runs in parallel with lint, can complete before lint
    fails, wasting compute.
    **Fix:** Add `needs: lint`.

13. **`docker-build` job in `ci.yml` duplicates `docker-publish.yml`**
    `ci.yml:208–256` runs on `push:main` and rebuilds images that
    `docker-publish.yml:4` also rebuilds on the same trigger. Two parallel
    builds, same images, double the storage churn.
    **Fix:** Remove the `docker-build` job from `ci.yml`; let `docker-publish.yml`
    own it.

14. **`KUBE_CONFIG_STAGING`/`KUBE_CONFIG_PRODUCTION` written to `/tmp/kubeconfig`**
    `deploy.yml:85, 124` — base64 decoded to a temp file. The runner is
    ephemeral, but this writes the cluster admin credential to disk
    unencrypted. Use `kubectl` env vars (`KUBECONFIG_DATA`) or the official
    `azure/k8s-set-context` action with a SP.
    **Fix:** Use `azure/k8s-set-context@<sha>` with workload identity, or at
    least delete the kubeconfig at job end.
