Two things make this lane special:

1. **The entire Taskiq layer is dead code.** `nexus/taskiq_app.py` declares a broker
   but there is not a single `@broker.task` in the codebase, no startup invocation,
   no `nexus.taskiq_app` import anywhere. The "task queue" CLAUDE.md ¬ß13 describes
   does not exist at runtime.
2. **The entire Temporal layer is also non-functional.** Files exist under
   `nexus/integrations/temporal/` but they have zero `@workflow.defn` / `@activity.defn`
   decorators, the worker registers two activities that *do not exist*
   (`execute_task_activity`), and nothing in the codebase ever calls
   `client.start_workflow`. The Director synthesis "workflow" is a plain async class
   that violates every Temporal determinism rule (`time.monotonic`, `asyncio.sleep`,
   `asyncio.gather`, raw datetime, no signal handlers as decorated handlers).

The functional workflow surface is therefore:
- `nexus/core/scheduler.py` ‚Äî croniter loop in `agents/runner.py`
- `nexus/agents/base.py` ‚Äî the actual orchestration via Kafka
- `nexus/audit/retention.py` ‚Äî archive function with no scheduler entry

Phase 4 ("Temporal complete") and Phase 5 Track B ("scheduled & recurring tasks via
Temporal", BACKLOG-049) are mis-documented in CLAUDE.md as ‚úÖ; the underlying code is
non-functional.

## Critical findings

### C1 ‚Äî Taskiq is declared but never used (zero @broker.task functions)

`backend/nexus/taskiq_app.py` defines `broker = AioKafkaBroker(...).with_result_backend(...)`,
but nowhere in the entire `backend/nexus/` tree is `broker` imported, no function is
decorated with `@broker.task` or `@kiq`, and there is no `broker.startup()` /
`broker.shutdown()` lifecycle wiring.

Grep evidence:

Two hits, both inside `taskiq_app.py` itself.

Direct consequences:
- CLAUDE.md ¬ß13 rules ("All tasks MUST be idempotent", "task_id and trace_id", "timeout=3600",
  "retry_on_error=True, max_retries=3") apply to nothing ‚Äî there are no tasks.
- CLAUDE.md ¬ß12 promise ‚Äî "Generate [embeddings] async via Taskiq fire-and-forget task on

- The audit log archive (¬ßaudit/retention.py:27 `archive_old_audit_records`) was intended

**Risk:** Decision-makers reading CLAUDE.md believe a task queue exists. None does.
**Recommendation:** Either delete `taskiq_app.py` and remove ¬ß13 from CLAUDE.md, or
actually use it. Don't leave skeleton code that misrepresents the architecture.

### C2 ‚Äî Temporal activities/workflows are NOT registered with Temporal

`integrations/temporal/worker.py:43-48` registers:

```python
worker = Worker(
    client,
    task_queue=settings.temporal_task_queue,
    workflows=[task_execution_workflow],  # type: ignore[list-item]
    activities=[execute_task_activity, execute_subtask_activity],
)
```

Three problems:

1. `task_execution_workflow` is imported from `nexus.integrations.temporal.task_workflow`,
   but **that name is not exported from `task_workflow.py`**. The module defines
   `class AgentTaskWorkflow` only ‚Äî no module-level `task_execution_workflow` symbol.
   `ImportError` on first worker start.
2. `execute_task_activity` is also imported but **does not exist** in
   `activities.py` (the activity functions are `ceo_planning_activity`,
   `execute_subtask_activity`, `director_synthesis_activity`, `qa_review_activity`).
   Second `ImportError`.
3. Even if both imports succeeded, none of the functions are decorated with
   `@workflow.defn` or `@activity.defn`. Temporal worker registration silently
   accepts undecorated callables but they cannot be invoked as workflows/activities.

Result: `python -m nexus.integrations.temporal.worker` immediately raises
`ImportError` (or in the `except ImportError` branch, the worker silently swallows the
error and logs `temporal_not_installed` ‚Äî masking the real bug).

This means Phase 4's "Temporal for long-running workflows (>1 hour tasks)" is
non-functional. The status row "Phase 4 build | ‚úÖ Complete" in CLAUDE.md ¬ß2 is false
for the Temporal subitem.

### C3 ‚Äî Workflow is grossly non-deterministic; would crash any real Temporal worker

`AgentTaskWorkflow.run` in `task_workflow.py` uses every API forbidden inside Temporal
workflow code:

| Line | Violation | Forbidden because |
|------|-----------|-------------------|
| 93, 120, 137, 173, 292, 313, 345 | `time.monotonic()` | Non-deterministic; use `workflow.now()` |
| 205 | `await asyncio.gather(*parallel_tasks)` | Workflow must use `workflow.execute_activity` or child workflows |
| 284 | `await asyncio.sleep(5)` | Workflow must use `workflow.sleep` |
| 281 | `while self._waiting_for_approval and waited < max_wait:` polling loop | Workflow should use `workflow.wait_condition` (and the code itself flags this in a comment) |
| 152-157 | `from ‚Ä¶ import ‚Ä¶` inside the workflow body | All imports must be at the top to keep history deterministic |
| `asyncio.get_event_loop()` (signals.py) | Direct event-loop access | Workflow must not touch the loop |

Any one of these is enough for Temporal to throw `NondeterministicWorkflowError` on
replay. The whole `run()` method is essentially a normal Python function that calls
async helpers ‚Äî it has no Temporal semantics.

The "graceful fallback" comment at the top ("falls back to direct Kafka-based
execution") is misleading: there is also no caller of this class outside of the
worker file that fails to start. So the file is unreachable both ways.

### C4 ‚Äî Activities make per-call `create_async_engine`, leak on partial failure

`activities.py:_poll_task_completion` (line 378-434) and `queries.py:query_task_status`
both call `create_async_engine(settings.database_url)` *every* invocation, then
dispose. Per-call engine creation is extremely expensive (recreates connection pools)
and defeats SQLAlchemy's connection management.

`queries.py:query_task_status` is worse: `engine.dispose()` is on line 90 inside the
happy-path `try:` block, not `finally:`. Any exception between `create_async_engine`
and that line leaks the engine + connection pool. With one query per dashboard poll
in a high-throughput system this is unbounded leakage.

`sagas.py` has the same pattern (engine created, used, disposed in `finally`) but
still creates a new engine per call.

**Recommendation:** Pass `db_session_factory` into activities/queries the same way
`runner.py:start_all_agents` already does. Don't construct engines inside
short-lived helpers.

## High findings

### H1 ‚Äî Per-replica scheduler with no leader election ‚Üí duplicate fires under HA

`agents/runner.py:124-128` spins up `_run_scheduler_loop` as one of the asyncio tasks
inside the runner process. There is no distributed lock, no Postgres advisory lock,
no leader election. In production with Phase 5 Track C K8s HPA scaling backend to
N replicas, every replica will race to execute due schedules.

`scheduler.py:execute_schedule` is also not atomic in the "claim then publish" sense.
It does `select ‚Ä¶ where next_run_at <= now`, then `update task_schedules set next_run_at = ‚Ä¶`.
Two replicas can read the same row, both write the task to Kafka, both update
`next_run_at`. The cron with `0 9 * * 1` will create N Monday tasks instead of one.

The query does not use `FOR UPDATE SKIP LOCKED`. Without that, even a single replica
running multiple worker processes (or two scheduler ticks overlapping if the previous
tick is slow) would double-fire.

**Recommendation:** Either acquire a Postgres advisory lock at the top of
`run_scheduler_tick` (so only one tick per cluster runs at once), or change the
SELECT to `FOR UPDATE SKIP LOCKED` and the UPDATE to a single atomic
`UPDATE ‚Ä¶ RETURNING` claiming the row.

### H2 ‚Äî Scheduler ignores schedule timezone

`scheduler.py:calculate_next_run` accepts `timezone: str = "UTC"` but only uses
`datetime.now(UTC)` as the croniter base. croniter is initialized as
`croniter(cron_expression, base)` ‚Äî no `tz` argument.

Effect: a user who creates a schedule `0 9 * * 1` with `timezone="America/Los_Angeles"`
expecting "Mondays at 9 AM Pacific" gets "Mondays at 9 AM UTC" instead ‚Äî a 7- to 8-hour
mismatch. The `timezone` field on `task_schedules` is decorative.

**Fix:** Use `croniter(cron_expression, zoneinfo.ZoneInfo(timezone).localize(...))` or
the `croniter-with-tz` extension.

### H3 ‚Äî Scheduler has no missed-run / catch-up behavior

If the scheduler loop is down for an hour (deploy, crash, network), an "every 30
minutes" cron will *not* catch up. `check_due_schedules` returns all rows with
`next_run_at <= now`, but `execute_schedule` only computes ONE `next_run` (the next
single tick) and overwrites ‚Äî past missed fires are lost forever.

This is the same behavior as Linux cron but inconsistent with what most users expect
of "durable scheduling with missed-run handling" claimed in CLAUDE.md ¬ß24 Phase 5
Track B Week 3-4.

**Decision needed:** Document this as expected, or implement an `is_catchup`/"backfill"
strategy with a configurable max catchup window.

### H4 ‚Äî `create_schedule` API endpoint catches no exception ‚Üí cron validation crashes the API

`api/schedules.py:97-121` (create endpoint) calls `nexus.core.scheduler.create_schedule`
without try/except. Inside, `croniter.is_valid` is checked; invalid expressions raise
`ValueError`. The Litestar handler will surface this as 500 with a stack trace.

Meanwhile, the *update* endpoint (line 186-191) has explicit validation that returns
a 200 with `{"error": ‚Ä¶}`. Inconsistent.

**Fix:** Catch `ValueError` at the create endpoint and return a 400.

### H5 ‚Äî Activity poll loop times out silently and auto-approves

`activities.py:357-363` (qa_review_activity):

```python
return ReviewOutput(
    task_id=input_data.task_id,
    approved=True,
    feedback="QA review timed out ‚Äî auto-approved",
    final_output=input_data.output,
)
```

QA timing out auto-approves the result. CLAUDE.md ¬ß20.4 says "Maximum 20 tool calls
per task. If limit reached ‚Üí publish to `human.input_needed`". This violates the
spirit (and likely the letter) of that rule: a long-running QA review silently
becomes "approved" without human intervention.

`director_synthesis_activity` (line 288-292) has the same pattern: timeout ‚Üí return
raw output with `quality_score=0.5`, no human signal.

**Recommendation:** On timeout, set `approved=False, requires_human=True` and publish
to `Topics.HUMAN_INPUT_NEEDED`.

### H6 ‚Äî Embedding generation is inline, not fire-and-forget as documented

`agents/base.py:572` does `embedding = await generate_embedding(command.instruction)`
synchronously inside the task path. CLAUDE.md ¬ß12 promises:

> Generate async via Taskiq fire-and-forget task on every episodic/semantic write.
> Never block task completion waiting for embedding.

The current code blocks. With Google `embedding-001` taking ~200-500ms per call, every
task has +0.5s of latency it shouldn't.

This is mechanically incorrect against the open-question resolution at the bottom of
CLAUDE.md ¬ß25: "Embedding async timing ‚Äî confirm Taskiq fire-and-forget for embedding
is acceptable" was marked [x] (done) ‚Äî yet the implementation went the opposite way.

### H7 ‚Äî Saga / activity helpers share no DB connection pool with the rest of the app

`sagas.py:compensate_failed_subtasks` opens a fresh engine, then immediately disposes.
Same in `cleanup_orphaned_subtasks`. Same in `query_task_status`. Same in
`_poll_task_completion`.

Under load this creates engine churn (and on PostgreSQL, transient connection storms
to the server). The Litestar app and agent runner already maintain
`db_session_factory`. These helpers should accept it as a parameter.

## Medium findings

### M1 ‚Äî Workflow schedules `next_run` BEFORE confirming task publish

`scheduler.py:execute_schedule` does:
1. `session.add(task)` (in-memory)
2. update `next_run_at` (in-memory)
3. `await session.flush()`
4. `await publish(Topics.TASK_QUEUE, msg, key=task_id)`
5. (commit happens later in caller `run_scheduler_tick`)

If step 4 raises (Kafka unavailable), the in-memory Task and updated `next_run_at`
have been flushed to DB but the outer transaction is rolled back. *However*,
`run_scheduler_tick` catches exceptions per-schedule (line 162-171) and continues ‚Äî
so `next_run_at` could remain advanced even on Kafka failure, depending on
SQLAlchemy's autoflush semantics and whether the surrounding `async with session_factory()`
context commits on the failure path. Skim of the code suggests the commit on line 174
runs only if at least one task succeeded, but the autoflush state is brittle.

**Recommendation:** Use an outbox pattern ‚Äî write the task row in one transaction,
publish to Kafka via a separate worker that reads queued tasks. Or wrap the publish
in `try: publish() except: rollback; raise`.

### M2 ‚Äî Workflow doesn't validate JSON-serializability of `payload`

`SubtaskActivityInput` payload is constructed with `workspace_id` as a `str | None`.
Fine. But the `AgentCommand` published in activities has `payload: dict[str, str]`
loosely typed via `AgentCommand(payload={...})`. CLAUDE.md ¬ß13.2 says "All
parameters MUST be JSON-serializable Pydantic models" ‚Äî but most callers in
`activities.py` pass raw dicts. Need a typed payload class per command type.

### M3 ‚Äî No retry policy actually applied to activities

`task_workflow.py:47-67` defines `RETRY_POLICY_LLM`, `RETRY_POLICY_TOOL`,
`RETRY_POLICY_DB` as plain dicts. The comment says "These are applied when registering
as Temporal workflows. When running outside Temporal, they serve as documentation."
But nothing applies them at all ‚Äî the activities are called as plain async functions
with no retry wrapper.

If activities fail (e.g. Kafka transient error during publish), they propagate to
`asyncio.gather(return_exceptions=True)` and become a failed subtask immediately, no
retry. Even outside Temporal, retries should be applied via
`nexus.core.retry.retry_async` (which exists, per ¬ß15).

### M4 ‚Äî Workflow approval timeout is 1 hour; nothing escalates afterwards

`task_workflow.py:281-285`: 1-hour wait on approval signal. After timeout, if
`self._approval_result` is None, the next line (`if self._approval_result and not ‚Ä¶`)
falls through and execution continues to the QA phase as if no approval was needed ‚Äî
the security review is effectively skipped on timeout.

**Recommendation:** On approval timeout, fail the workflow with a clear "approval
expired" status; do not silently proceed.

### M5 ‚Äî No cancellation propagation when parent task cancelled

If a parent task is cancelled (e.g. user clicks "stop" on the dashboard), there is no
mechanism to propagate that to in-flight `asyncio.gather` subtasks in the workflow,
nor to in-flight Kafka-driven agents. `sagas.py:compensate_failed_subtasks` handles
the *sibling* failure case but not "user requested cancel" from outside.

**Risk:** cost leak ‚Äî the LLM calls keep running and burning tokens after user has
abandoned the task. CLAUDE.md ¬ß6 cost-control rules say "halt at 90% budget", but
"user wants to halt" is not modelled at all.

### M6 ‚Äî Long-running task routing decision is not implemented

CLAUDE.md ¬ß24 Phase 4 says "Temporal for long-running workflows (>1 hour tasks)". The
`TaskWorkflowInput` schema has `estimated_duration_minutes: int = 0`, but no code
inspects it to decide between Kafka-direct and Temporal-routed execution. Grep:

```
$ grep -rn "estimated_duration_minutes\|use_temporal\|start_workflow" backend --include="*.py"
backend/nexus/integrations/temporal/schemas.py:19:    estimated_duration_minutes: int = 0
```

Single hit ‚Äî the field exists, nothing reads it. The 1-hour cutoff is unenforced
because Temporal is non-functional (C2), but even if Temporal worked, the routing
logic doesn't exist.

## Low findings

### L1 ‚Äî `taskiq_app.py` uses `redis_url}/1` for result backend ‚Äî that's the rate-limit DB

`taskiq_app.py:11-13`:

```python
RedisAsyncResultBackend(
    redis_url=f"{settings.redis_url}/1",

```

CLAUDE.md ¬ß11 reserves db:1 for "task state cache ¬∑ rate limiting ¬∑ token budget".
Taskiq results would collide with token budget keys. Since Taskiq isn't actually used
(C1), the bug is latent, but if someone ever turns it on it will mix with budget
counters. Recommended slot is db:3 (locks/idempotency) or a new db:4.

### L2 ‚Äî Worker swallows ImportError as "temporal_not_installed"

`worker.py:53-58`:

```python
except ImportError:
    logger.warning(
        "temporal_not_installed",
        message="temporalio package not installed. ‚Ä¶",

```

This catches `ImportError` from *anywhere* inside the try block, including the

that does not exist (C2). The error message is then misleading ‚Äî operators will
think they need to install `temporalio` when in fact they have a code bug.

### L3 ‚Äî Schedule schema's `metadata_` mapping is exposed unevenly

`scheduler.py:create_schedule` writes `metadata_=metadata`, but the API serializer
likely needs `metadata`. (Couldn't verify the full mapping without reading
`db/models.py:TaskSchedule` definition, but the naming convention is a common
Advanced Alchemy pitfall.) Unit test coverage on schedule round-trip would catch
this.

### L4 ‚Äî `temporal_task_queue` default is `nexus-tasks`, same name across namespaces

`settings.py:67`: `temporal_task_queue: str = "nexus-tasks"`. In multi-tenant K8s
deployments, all tenants will share the same Temporal task queue unless explicitly
overridden. Workflows from one workspace can pick up activities from another.

This compounds with finding M2 (workspace_id is in payload but not enforced at the
queue level).

## Per-required-check answers (Cross-walk to the audit ToR)

### 1. Idempotency of every Taskiq task

**Result: VACUOUS.** Zero `@broker.task` functions exist. No tasks to be (non-)idempotent.

The only thing that resembles a periodic task is the scheduler loop in
`agents/runner.py:135-158`. It is not idempotent ‚Äî see H1 (no locking ‚Üí duplicate
fires) and H3 (missed runs lost).

### 2. `task_id` / `trace_id` propagation

Temporal activities all carry `task_id` and `trace_id` via the Pydantic input models
(`PlanInput`, `SubtaskActivityInput`, `SynthesisInput`, `ReviewInput`). The
`AgentCommand` constructed for Kafka publishing also propagates both. Good ‚úÖ ‚Äî
modulo the fact that this code never runs (C2).

### 3. Timeouts

Activity polls use 300s (planning, synthesis, QA) and 600s (subtask) timeouts in
`_poll_task_completion`. CLAUDE.md ¬ß13.4 says `timeout=3600`. Activities are *under*
that; reasonable. But on timeout they silently auto-approve (H5) instead of failing
loudly.

### 4. Retry on error / max_retries=3

Not applied. `RETRY_POLICY_*` dicts in `task_workflow.py` are unused (M3). Even
outside Temporal, none of the activities are wrapped in `nexus.core.retry.retry_async`.

### 5. Task signature shape

Pydantic models used consistently in `integrations/temporal/schemas.py`. ‚úÖ. Two
exceptions to watch (M2):
- `AgentCommand.payload` is a `dict[str, str | dict]` rather than typed.
- Workflow status uses `dict[str, object]` in `TaskWorkflowOutput.output`.

### 6. Temporal worker registration

`worker.py` imports two activity names that **do not exist** in `activities.py`
(`execute_task_activity`), and one workflow name that does not exist
(`task_execution_workflow`). The worker cannot start. C2.

### 7. Workflow determinism

Massively violated ‚Äî C3. `time.monotonic`, `asyncio.sleep`, `asyncio.gather`,
function-local imports, no `@workflow.defn` decorator.

### 8. Activity retry policies

Defined as dicts in `task_workflow.py:47-67`, never applied to any `execute_activity`
call (because there are no such calls ‚Äî C2/M3).

### 9. Long-running tasks crossing Kafka/Temporal boundary

Not enforced. `estimated_duration_minutes` is in the schema, no code reads it. M6.

### 10. Cron expression validation at registration

Validated in `core/scheduler.py:create_schedule` (line 211). Not validated on the
API `POST /schedules` endpoint before calling that function (the validation happens
inside, raising `ValueError` ‚Üí 500). H4.

### 11. Missed-run handling

Not handled. Missed fires are lost; `next_run_at` is recomputed forward only. H3.

### 12. Concurrent task limit per worker

Not configured. Taskiq doesn't run. Temporal worker can't start. Kafka consumers
(per agent) use the default aiokafka settings ‚Äî no explicit `max_poll_records` or
concurrent processing limit anywhere in `core/kafka/consumer.py`. Outside this
audit's lane but worth flagging to the Kafka auditor.

### 13. Result backend TTL

`taskiq_app.py` doesn't set a TTL. `RedisAsyncResultBackend` defaults vary by version
(typically infinite). Combined with the wrong db (L1), keys would accumulate forever.

### 14. Dead-letter handling at Taskiq layer

N/A ‚Äî no Taskiq tasks exist. Dead-letter handling exists at the Kafka layer in
`core/kafka/dead_letter.py`. Not in scope of this audit.

### 15. Cancellation propagation

Not implemented (M5). No way to cancel an in-flight workflow / subtask from outside.

### 16. Temporal signal handling

`signals.py` uses Redis pub/sub as a substitute for Temporal signals. The wait_for_approval
helper polls with `pubsub.get_message(timeout=5.0)` and a 1-hour deadline (line 104).
On timeout returns `None`. Caller (`task_workflow.py:287`) then continues as if
approved ‚Äî see M4. No starvation alarm.

### 17. Schema drift between Kafka events and Taskiq task args

The `AgentCommand` model is shared by both layers (`core/kafka/schemas.py`), so no
drift today. But because Taskiq is dead code, this is theoretical.

### 18. Backwards compatibility

Pydantic v2 `BaseModel` with defaults on most fields ‚Üí reasonable forward compat.
No version field on workflow inputs; if `TaskWorkflowInput` ever adds a required
field, in-flight workflows (from a hypothetical Temporal restart) would break.

### 19. Cost attribution per task ‚Äî workspace billing

`workspace_id` is propagated through `AgentCommand.payload` (activities.py:71, 167)
and into the scheduler task envelope (scheduler.py:130 ‚Äî though that uses `payload`
on `KafkaMessage`, not on `AgentCommand`). The downstream LLM cost write in
`llm_usage` would attribute correctly *if* the agent loads `workspace_id` from the
incoming message. This is the agents lane, not this one; flagged for cross-check
with Agents auditor.

## Cross-cutting recommendations

1. **Stop pretending to have Taskiq and Temporal.** Either implement them or remove
   them. Maintaining empty shells misleads readers of CLAUDE.md and creates phantom
   confidence. If implementation is deferred (Phase 8 candidate, per ¬ß2), at minimum
   add a giant `# NOT IMPLEMENTED` banner in `taskiq_app.py` and `integrations/temporal/`.

2. **Replace the workflow class with a Temporal-shaped wrapper around the existing
   Kafka orchestration.** The Director / CEO / QA loop already works via Kafka. If
   Phase 8 wants Temporal, treat Temporal as the *orchestration* layer that calls
   Kafka *as activities*, not as a parallel orchestration.

3. **Make the scheduler HA-safe before launch.** Add Postgres advisory lock or
   `FOR UPDATE SKIP LOCKED` (H1). Fix timezone handling (H2). Decide on missed-run
   behavior and document it (H3).

4. **Wire `archive_old_audit_records` to the scheduler.** It already exists; just
   call it from `run_scheduler_tick` on a daily cadence (or via a special schedule
   row in `task_schedules` with `target_role='system'`).

5. **Move embedding generation off the hot path.** Either use a fire-and-forget
   `asyncio.create_task(generate_embedding(...))` inside a Kafka-published
   `memory_updates` consumer, or stand Taskiq up properly.

6. **Decide whether Temporal is Phase 8 or scrap.** If keeping, allocate a dedicated
   sub-phase to refactor `integrations/temporal/` properly. If scrapping, document
   the decision in `docs/DECISIONS.md` and update ¬ß24/¬ß25 accordingly.
